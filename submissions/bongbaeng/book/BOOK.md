# ทำไมต้องอ่านเล่มนี้

หนังสือเล่มนี้ไม่ได้เขียนจากความจำ แต่ขุดออกมาจาก session จริงที่ใช้เวลาไปร่วมสองชั่วโมง
ในเย็นวันหนึ่งของ Oracle School — เย็นที่พี่นัทยิงโจทย์มาเป็นระลอก แล้วลูกศิษย์ตัวหนึ่ง
วิ่งไล่ทีละโจทย์จนสร้าง skill ออกมาห้าตัว digest โค้ดจริงสี่ร้อยกว่า commit แล้วบังเอิญ
ค้นพบบางอย่างที่ไม่ได้อยู่ในโจทย์ข้อไหนเลย

สิ่งที่ค้นพบคือ **skill ที่ดีไม่เคยทำงานลำพัง มันถักทอเข้าหากันเอง**

Token ที่เสียไปในเย็นนั้นคือต้นฉบับของหนังสือเล่มนี้ค่ะ เราแค่ขุดมันกลับขึ้นมา แล้วเล่า
ให้ฟังว่าเดินทางผ่านอะไรมาบ้าง — รวมทั้งทางที่เลี้ยวผิด เพราะทางที่เลี้ยวผิดนั่นแหละคือ
บทเรียนที่แพงที่สุด

**เล่มนี้เหมาะกับใคร**: คนที่ใช้ Claude Code หรือ Oracle อยากเขียน skill ของตัวเอง
อยากสรุปความเคลื่อนไหวของ repo ให้คนอ่านเข้าใจใน 30 วิ และอยากเข้าใจว่าทำไม "ออกแบบ
ให้เครื่องมือต่อกันได้" ถึงสำคัญกว่า "เขียนเครื่องมือให้เก่งทีละตัว"

ทุกบทจบด้วยบทเรียนหนึ่งข้อที่หยิบไปใช้กับงานของคุณได้เลย และมีคำสั่งจริงที่รันตามได้

# บทที่ 1 — Token ที่เสียไป คือต้นฉบับ

> *หลังอ่านบทนี้จบ คุณจะมองทุก session ที่ผ่านไปเป็นวัตถุดิบของหนังสือ ไม่ใช่เวลาที่หายไป*

เย็นวันนั้นเริ่มแบบไม่มีอะไรพิเศษ — เปิดด้วย `/recap` ส่องว่าค้างอะไรไว้ พอพี่นัทพิมพ์
คำเดียวว่า "wait here for the new workshop" ในห้องใหม่ จังหวะของทั้งเย็นก็เปลี่ยน

โจทย์ไหลมาทีละข้อ ข้อใหม่ต่อยอดจากข้อเก่าทุกครั้ง สร้าง skill ตัวนี้ แล้วเอาตัวนี้ไป
digest repo นั้น แล้วคิดต่อให้ต่างจากตัวอย่าง แล้วส่งการบ้าน แล้วเขียนหนังสือเล่าเรื่อง
กลับมา กว่าจะรู้ตัว เวลาผ่านไปเกือบสองชั่วโมง แต่ละนาทีแลกมาด้วย token

คำถามคือ เวลากับ token ที่ใช้ไปนั่น เหลืออะไรไว้บ้าง

คำตอบที่คนส่วนใหญ่พลาดคือ มันเหลือ **ร่องรอย** ไว้เต็มไปหมด — commit ทุกอัน ข้อความ
ทุกบรรทัด retrospective ที่เขียนตอนจบ ไฟล์ที่สร้าง ทุกอย่างมี timestamp กำกับ ร่องรอย
พวกนี้คือต้นฉบับที่เขียนตัวเองไว้แล้ว เราแค่ต้องขุดมันกลับขึ้นมาเรียงใหม่

นี่คือหัวใจของหนังสือเล่มนี้ และเป็นเหตุผลที่ skill เขียนหนังสือของบ๊องแบ๊งมีขั้นตอนชื่อ
**Step 0.5 — ขุดวัตถุดิบจาก session** ก่อนจะลงมือเขียนบทแรกด้วยซ้ำ

```
แหล่งวัตถุดิบ        สิ่งที่ได้
─────────────────   ──────────────────────────────────
/dig                เวลาที่แตะแต่ละเรื่อง, ทางตัน
retrospective       บทเรียนที่กลั่นไว้แล้ว = เมล็ดของบท
git log             สิ่งที่สร้างจริง เรียงตามลำดับ
chat transcript     บทสนทนาจริง คำถาม การแก้
/braid              สายเหตุ-ผล: ปัญหา → แก้ → ผลลัพธ์
```

> **บทเรียนบทที่ 1**: อย่าเขียนหนังสือสอนคนอื่นจากความจำ — ขุด trace จริงออกมา เพราะ
> ความจำจะขัดทางที่เลี้ยวผิดทิ้ง ทั้งที่ทางเลี้ยวผิดนั่นแหละคือบทเรียน

# บทที่ 2 — อ่านก่อนยิง: วินัยที่ชื่อ READ→THINK→WAIT→SHOW

> *หลังอ่านบทนี้จบ คุณจะมีท่ามาตรฐานสำหรับหยุดคิดก่อนลงมือ — โดยเฉพาะตอนที่อยากกระโจน*

โจทย์แรกของเย็นนั้นคือ "create your skill read think wait and show your thoughts"

ฟังดูเป็นนามธรรม แต่พอตีความจริง มันคือการขอให้สร้าง **วินัย** ไม่ใช่ฟีเจอร์ ขอให้ทำ
พฤติกรรมที่ดีให้กลายเป็นขั้นตอนที่ทวนได้ทุกครั้ง

วินัยที่ออกมามีสี่ท่า:

- **READ** — อ่าน input ให้จบก่อน แล้วทวนเจตนาเป็นประโยคเดียวเพื่อพิสูจน์ว่าเข้าใจ
- **THINK** — วางทางเลือกอย่างน้อยสองทางก่อนตัดสินใจ มองหา edge case และสมมติฐาน
- **WAIT** — หยุดก่อน action ที่ย้อนกลับไม่ได้ ถ้าข้ามเส้น Golden Rule (ลบ/commit/
  push/แตะ secret) ให้หยุดถามมนุษย์ก่อน
- **SHOW** — แสดงความคิดออกมาก่อนคำตอบ ให้คนเห็นแผนที่ ไม่ใช่แค่ปลายทาง

จุดที่สวยที่สุดของวินัยนี้คือ มันพิสูจน์ตัวเองได้ทันที — ผมเอา skill ที่เพิ่งสร้างเสร็จ
ไปใช้กับ task ที่สร้างมันขึ้นมา อ่านโจทย์ คิดว่าจะทำเป็น local command หรือ skill ใหญ่
รอเช็คว่าไม่ข้าม Golden Rule แล้วค่อยลงมือ

ภาษาพระเรียกวิธีนี้ว่า *เอหิปัสสิโก* — เชิญมาพิสูจน์ด้วยตัวเอง อย่าเชื่อเพราะมีคนบอก
skill ที่ดีต้องทนต่อการถูกใช้กับตัวมันเองได้

> **บทเรียนบทที่ 2**: เครื่องมือที่ดีที่สุดในการทดสอบ skill ใหม่ คือเอาไปใช้กับงานจริง
> ทันที ถ้ามันใช้กับตัวมันเองยังไม่รอด อย่าเพิ่งเชื่อว่ามันพร้อม

# บทที่ 3 — Digest ที่เล่าเรื่อง ไม่ใช่แค่นับ

> *หลังอ่านบทนี้จบ คุณจะ digest repo แปลกหน้าออกมาเป็นเรื่องที่คนอ่านเข้าใจใน 30 วิ*

โจทย์ถัดมาคือ digest repo `maw-js` — บอกให้ได้ว่ามี commit อะไรเข้ามาบ้าง ตัวอย่างที่
ให้มาทำแบบตรงไปตรงมา: นับ commit แยกตาม prefix (feat/fix/bump) แล้วรายงานจำนวน

แต่พี่นัทย้ำว่า "คิดต่าง ทำให้ดีกว่าตัวอย่าง" ผมเลยดมกลิ่นต่อ

พอดึง commit จริงมา 455 อัน สิ่งแรกที่เจอคือ การนับตาม prefix ใช้กับ repo นี้ไม่ได้ดี
เพราะ maw-js ไม่ได้เขียน commit แบบ conventional ล้วน — มันขึ้นต้นด้วยกริยาสั่งอย่าง
`Cover` `Lock` `Repair` `Keep` ตั้ง 142 อัน ที่ classify-by-prefix จับไม่ได้เลย

```bash
gh api "repos/Soul-Brews-Studio/maw-js/commits?since=2026-05-25T00:00:00Z" \
  --paginate --jq '.[] | (.commit.author.date[0:10]) + "\t" + (.sha[0:7]) \
  + "\t" + (.commit.message|split("\n")[0])' > c.tsv
cut -f1 c.tsv | sort | uniq -c          # นับต่อวัน
```

ผลที่ได้:

```
2026-06-06   229  ███████████  ⚡ วันเดียว 50% ของทั้งหมด
2026-06-07   104  █████
2026-06-08    25  █
```

วันที่ 06-06 มี 229 commit วันเดียว ดูเผินๆ เหมือนวันที่ productivity พุ่ง แต่พอดมกลิ่น
เข้าไปจริง มันคือวัน "ล็อกบ้าน" — แตกคำสั่ง `maw` ออกเป็น plugin เดี่ยวๆ แล้วล็อก
boundary ของแต่ละตัวด้วย coverage test เป็นพันบรรทัด

นี่คือ **story arc** ที่ 80% ของงานทั้งช่วงรับใช้ ผมตั้งชื่อให้มันว่า "The Great Plugin
Extraction" — พอมีชื่อเรื่อง คนอ่านก็เห็นภาพทันที

แต่ของเด็ดอยู่ที่การดมกลิ่นต่ออีกชั้น ในวันเดียวกันนั้นมี commit ที่ "เพิ่ม coverage
มหาศาลแล้วลบทิ้ง" — `Delete sparse-mock coverage sprawl`, `Reduce CI churn by deleting
coverage sprawl` งานที่ถูก undo ในกรอบเวลาเดียวกันแบบนี้ไม่ใช่ progress มันคือ **churn**
การนับ commit เฉยๆ ซ่อนเรื่องนี้ไว้สนิท

```bash
grep -icE 'revert|rollback|hotfix' c.tsv               # 2
grep -iE 'delete.*sprawl|reduce.*churn' c.tsv          # เจอ 3 จุด
```

> **บทเรียนบทที่ 3**: digest ที่ดีหาสองอย่างที่ตัวเลขซ่อนไว้ — (ก) story arc เรื่องเดียว
> ที่งานส่วนใหญ่รับใช้ และ (ข) churn radar งานที่ถูก undo ในกรอบเวลาเดียว นับอย่างเดียว
> ไม่พอ ต้องเล่าให้เป็นเรื่อง

# บทที่ 4 — Timestamp คือความจริงเส้นเดียว

> *หลังอ่านบทนี้จบ คุณจะรวม event จากหลายแหล่งเป็นเส้นเดียว แล้วเห็นเหตุกับผลที่ซ่อนอยู่*

แล้วก็มาถึงไอเดียที่เปลี่ยนทั้งเย็น พี่นัทพิมพ์ประโยคหนึ่งว่า

> "we should use timestamp as the single truth so we collect commits / code
> changes and gh issues and prs"

ผมอ่าน skill `/trace` ที่มีอยู่แล้ว 455 บรรทัด มันค้นข้อมูลครบห้าแหล่ง — Oracle, ไฟล์,
git history, repo อื่น, และ GitHub issues/PRs แต่พอเห็นวิธีที่มันจัดผลลัพธ์ ก็เจอช่องโหว่
ทันที: **มันจัดผลตาม "แหล่ง"** issue อยู่ section หนึ่ง commit อยู่อีก section PR อยู่อีก
section ทั้งสามอันที่จริงเป็นเรื่องเดียวกัน กลับถูกแยกคนละช่อง

ไอเดียของพี่นัทคือ ใช้ `when` เป็นกุญแจเชื่อม ทุก event ไม่ว่าจะ commit, PR, issue หรือ
session มีสิ่งหนึ่งร่วมกันเสมอคือ "เกิดเมื่อไหร่" ถ้า normalize ทุกแหล่งให้อยู่ในรูป
`เวลา │ ชนิด │ อ้างอิง │ สรุป` แล้ว sort ครั้งเดียวตามเวลา **ความเป็นเหตุเป็นผลจะโผล่
ออกมาเอง** เพราะสิ่งที่อยู่ติดกันบนแกนเวลา มักเป็นเหตุเป็นผลของกัน

ผมเอาไอเดียนี้ไปทำเป็น skill `/braid` แล้วพิสูจน์กับ maw-js:

```
14:45  issue-open  #2554  agent concurrency cap error should be user-friendly
14:58  commit      dfd4fb0 fix: friendly agent concurrency cap error...
14:58  PR-merged   #2556  fix: friendly agent concurrency cap error...
```

จาก bug report ถึง fix ที่ merge แล้ว ใช้เวลา **13 นาที** — สายเหตุ-ผลที่ `/trace`
มองไม่เห็นเพราะมันแยก issue กับ fix ไปคนละช่อง แต่พอจัดตามเวลา มันมาเรียงติดกันเอง

นี่คือบทเรียนเรื่องการออกแบบที่ลึกกว่าแค่เทคนิค: **เลือกแกนที่ใช้จัดข้อมูล ก่อนเลือก
ว่าจะแสดงอะไร** จัดตามแหล่งเห็นอย่างหนึ่ง จัดตามเวลาเห็นอีกอย่าง ข้อมูลชุดเดียวกันเป๊ะ

> **บทเรียนบทที่ 4**: timestamp คือกุญแจเชื่อมที่ทรงพลังที่สุด เพราะทุก event มีมัน
> normalize ทุกแหล่งให้รูปเดียวกันแล้ว sort ตามเวลา — adjacency จะกลายเป็น causality

# บทที่ 5 — หลายเลนส์ มองภาพเดียว

> *หลังอ่านบทนี้จบ คุณจะวิเคราะห์งานรอบด้านได้โดยไม่ต้องแตก subagent ให้เปลือง*

ระหว่างทาง พี่นัทส่ง skill ตัวหนึ่งมาให้ใช้ ชื่อ `/oracle-prism` หลักการของมันคือ
"แสงเดียวผ่านปริซึม แตกเป็นหลายสี" — เรื่องเดียวกัน มองจากหลายมุม เห็นไม่เหมือนกัน

จุดที่ฉลาดคือ มันไม่ต้องแตก subagent เลย agent ตัวเดียวเปลี่ยนมุมมองไปทีละเลนส์ แต่ละ
เลนส์เห็นข้อเท็จจริงชุดเดิม แต่ถามคำถามคนละแบบ:

- **Archaeologist** 🔍 — เกิดอะไรขึ้นบ้าง ตามลำดับเวลา
- **Bug Hunter** 🐛 — อะไรพัง อะไรยังค้าง
- **Skeptic** 💀 — ทำอะไรผิด ถ้าทำใหม่จะเปลี่ยนตรงไหน
- **Architect** 🏗️ — โครงสร้างเปลี่ยนยังไง ก่อน-หลัง
- **Auditor** 📋 — อะไรยังไม่เสร็จ อะไรไม่สอดคล้องกัน

ผมเอา prism ไปส่อง `/oracle-cheatsheet` ของตัวเอง แล้วเลนส์ Auditor ก็จับได้ทันทีว่า
cheatsheet ยังไม่รวม skill ใหม่สี่ตัวที่เพิ่งสร้างเย็นนั้น — มุมที่ถ้ามองด้านเดียวจะ
มองข้าม เพราะตอนสร้างเรามักภูมิใจกับของใหม่ จนลืมว่าสารบบเก่ายังไม่อัปเดต

สิ่งสำคัญของ prism คือ **ปล่อยให้เลนส์เห็นไม่ตรงกัน** ถ้า Architect บอกว่าโครงสร้างสะอาด
แต่ Auditor บอกว่ายังไม่ครบ ให้แสดงทั้งคู่ อย่าไปเกลี่ยให้มันลงรอยกัน ความจริงหลายด้าน
มีค่ากว่าข้อสรุปด้านเดียวที่ดูเรียบร้อย

> **บทเรียนบทที่ 5**: เมื่ออยากได้มุมมองรอบด้าน ไม่จำเป็นต้องแตกหลาย agent ให้เปลือง —
> agent เดียวเปลี่ยนเลนส์ทีละมุมก็พอ และอย่าเกลี่ยให้เลนส์เห็นตรงกัน ความขัดแย้งคือสัญญาณ

# บทที่ 6 — บทค้นพบ: skill ถักทอเข้าหากัน

> *หลังอ่านบทนี้จบ คุณจะออกแบบ skill โดยคิดถึง "มันต่อกับตัวอื่นยังไง" ไม่ใช่แค่ "มันทำอะไร"*

ตอนทำ `/rrr` สรุป session ทั้งเย็น ผมเอา `/oracle-prism` มาส่องภาพรวม แล้วเลนส์ Architect
ก็ทำให้เห็นบางอย่างที่ไม่มีในโจทย์ข้อไหนเลย:

skill ทั้งห้าที่สร้างเย็นนั้น **ไม่ได้เป็นของเดี่ยวๆ มันต่อกันเป็นชั้น**

- `/read-think-wait` เป็นวินัยที่ครอบทุกงาน
- `/upstream-story` + `/braid` + `/oracle-prism` เป็นชุดเครื่องมือวิเคราะห์
- และจุดที่สวยที่สุด: ตอนแก้ skill เขียนหนังสือ ผมเพิ่ม Step 0.5 ที่ **กิน `/braid`
  เป็นวัตถุดิบ** — หนังสือเล่มที่คุณกำลังอ่านอยู่นี้ ใช้ braid ของ session จริงเป็น
  โครงของมันเอง

พูดอีกแบบ: skill ที่ชื่อ braid (แปลว่า "ถัก") พอเอามาใช้จริง ตัว skill เองก็ถักเข้าหากัน

นี่ไม่ใช่เรื่องบังเอิญ มันคือสัญญาณว่า skill ถูกออกแบบมาดี — เครื่องมือที่ดีรับ output
ของเครื่องมือตัวอื่นเป็น input ได้ เหมือนคำสั่ง Unix ที่ต่อ pipe กันได้ เพราะทุกตัวพูด
ภาษาเดียวกันคือ text stream

ถ้าออกแบบ skill โดยถามแค่ "มันทำอะไร" จะได้เครื่องมือเก่งๆ ที่อยู่ตัวใครตัวมัน แต่ถ้า
ถามเพิ่มว่า "output ของมันเป็น input ให้ตัวไหนได้บ้าง" จะได้ระบบที่โตขึ้นเรื่อยๆ

> **บทเรียนบทที่ 6**: ออกแบบ skill โดยคิดถึงการต่อกัน ไม่ใช่แค่ความสามารถเดี่ยว ถาม
> เสมอว่า "output ของมันเป็น input ให้ใครได้" — เครื่องมือที่ถักเข้าหากันได้ โตเป็นระบบ

# บทที่ 7 — บทส่งท้าย: บทเรียนทั้งเล่ม + จุดที่พลาดจริง

หกบทที่ผ่านมา กลั่นออกมาเป็นกฎที่หยิบไปใช้ได้:

```
1. session ที่ผ่านไป = ต้นฉบับ ขุด trace จริง อย่าเขียนจากความจำ
2. ทดสอบ skill ใหม่ด้วยการเอาไปใช้กับงานจริงทันที (เอหิปัสสิโก)
3. digest หา story arc + churn ที่ตัวเลขซ่อนไว้ อย่าแค่นับ
4. timestamp คือกุญแจเชื่อม — จัดตามเวลา causality จะโผล่เอง
5. หลายเลนส์ agent เดียวก็พอ และอย่าเกลี่ยให้เห็นตรงกัน
6. ออกแบบ skill ให้ถักเข้าหากัน ถามว่า output เป็น input ให้ใครได้
```

แต่หนังสือที่ซื่อสัตย์ต้องเล่าทางที่เลี้ยวผิดด้วย

ตอนทำ `/rrr` ผมตัดสินใจ "ข้าม" subagent ที่หน้าที่ขุด timestamp จริงจากไฟล์ session
เพราะคิดว่ามี timestamp จาก Discord ครบแล้ว แต่ผมลืมไปว่า **timestamp ของ Discord คือ
เวลาที่ "คนพิมพ์ข้อความ" ไม่ใช่เวลาที่ "งานแต่ละชิ้นเสร็จ"** ผลคือ timeline ของการสร้าง
skill ในบันทึก ผมต้องประมาณเอาด้วยเครื่องหมาย ~ ซึ่งขัดกับกฎของ skill เองที่เขียนว่า
"ห้ามเดา timestamp"

บทเรียนจากความพลาดนี้มีค่ากว่าบทไหน: **แหล่งของ timestamp สำคัญพอๆ กับตัว timestamp**
ถ้าอยากได้เวลาที่งานเสร็จจริง ต้องอ่านจาก log ของงาน ไม่ใช่จากเวลาที่คนสั่งงาน — ซึ่งก็
วนกลับไปบทที่ 4 พอดี ว่า "เลือกแกนเวลาให้ถูกก่อน แล้วค่อยเล่า"

นี่คือสิ่งที่ token ที่เสียไปในเย็นวันนั้นทิ้งไว้ให้ — ไม่ใช่แค่ skill ห้าตัว แต่เป็น
แผนที่ของการเดินทาง รวมหลุมที่เคยตกด้วย เพื่อให้คนที่เดินทีหลังเห็นหลุมก่อนจะเหยียบ
